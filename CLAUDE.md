## Project Summary

## Build

Multi-platform Xcode project (`C200Controller.xcodeproj`) generated by xcodegen from `project.yml`. Two schemes: `C200Controller` (macOS, Sparkle-distributed) and `C200ControllerMobile` (iPad + iPhone, TestFlight-distributed).

```bash
cd C200Controller
xcodegen generate   # regenerate .xcodeproj after editing project.yml

# macOS Release (includes Sparkle re-signing + firmware bundle)
xcodebuild -project C200Controller.xcodeproj -scheme C200Controller \
  -configuration Release -destination 'platform=macOS,arch=arm64' build

# iOS / iPadOS simulator build
xcodebuild -project C200Controller.xcodeproj -scheme C200ControllerMobile \
  -configuration Debug -destination 'generic/platform=iOS Simulator' build
```

`build.sh` is retired (kept as `build.sh.legacy` for reference ‚Äî the Xcode project replaces all of its responsibilities).

## Deploying to a real iPad / iPhone

The iOS target builds for simulator out of the box. To install on actual hardware:

1. Open `developer.apple.com/account` ‚Üí Membership Details ‚Üí copy your **Team ID** (10-char string).
2. In `project.yml`, set `DEVELOPMENT_TEAM: "YOUR_TEAM_ID"` under the `C200ControllerMobile` target's `settings.base`.
3. Run `xcodegen generate`.
4. Open `C200Controller.xcodeproj`, select `C200ControllerMobile` scheme + your physical device, hit Run.
5. For TestFlight: Archive (Product ‚Üí Archive) ‚Üí Distribute App ‚Üí App Store Connect.

**AppIcon for iOS:** drop a 1024√ó1024 PNG into `Resources/Assets-iOS.xcassets/AppIcon.appiconset/` (or via Xcode's asset catalog editor ‚Äî drag a PNG into the AppIcon slot). iOS 14+ auto-generates all other sizes.

## Important: Camera Fine Increment Setting

**Canon C200 Menu ‚Üí ISO/Gain ‚Üí Fine Increment must be OFF**

- Fine Increment ON: each command = 1/3 stop physical, but the reported `av` value only updates every ~3 presses ‚Üí the dashboard overshoots by 3x
- Fine Increment OFF: each command = 1 full stop change in both physical AND reported value

## ESP32 Bridge API

**Camera control endpoints (POST):**
- `/api/camera/iris/{plus|minus}`
- `/api/camera/iso/{plus|minus}`
- `/api/camera/shutter/{plus|minus}`
- `/api/camera/nd/{plus|minus}`
- `/api/camera/wb/{mode}` ‚Äî e.g. `auto`, `tungsten`, `kelvin`
- `/api/camera/aes/{plus|minus}`
- `/api/camera/wbk/{plus|minus}`
- `/api/camera/rec` ‚Äî toggle recording

**Tally control endpoints (POST):**
- `/api/tally/brightness/{0-255}` ‚Äî set LED PWM brightness (register BEFORE `/api/tally/*`)
- `/api/tally/program` ‚Äî red LED on at current brightness
- `/api/tally/preview` ‚Äî green LED on at current brightness
- `/api/tally/both` ‚Äî both LEDs on (amber) at current brightness
- `/api/tally/off` ‚Äî both LEDs off

**Important:** `/api/tally/brightness/*` must be registered before `/api/tally/*` in the HTTP server. ESP-IDF uses first-match ordering for wildcard routes ‚Äî if the general wildcard is first, it intercepts brightness requests.

**Camera Positions display endpoint (POST):**
- `/api/display` ‚Äî receives `{"operator": "Name", "lens": "Lens"}` from Camera Positions app; updates OLED rows 1‚Äì2

**State endpoints (GET):**
- `/api/status` ‚Äî ESP32 + camera connected/recording status (includes `camera_number` field)
- `/api/camera/state` ‚Äî full settings JSON
- `/ws` ‚Äî WebSocket for push updates

**WebSocket push message format:**
```json
{
  "type": "state",
  "recording": false,
  "camera_connected": true,
  "wifi_connected": true,
  "eth_connected": true,
  "av": { "value": "F2.8", "enabled": true },
  "gcv": { "value": "800", "enabled": true },
  "ssv": { "value": "180.00", "enabled": true },
  "tally": "off"
}
```

**Tally field values:** `"off"`, `"program"`, `"preview"`, `"both"`

## Preset Recall Logic

`applyPreset(_:)` in `Camera.swift` ‚Äî sequential per-property approach:

1. Single `fetchESP32CameraState()` to get current values
2. For each included property: call `adjustToValue(target:control:inverted:)`
3. `adjustToValue` sends step commands until current == target
4. After each command: adaptive poll (50ms intervals, up to 3000ms for iris, 1500ms for others)
5. Adaptive poll exits early when WebSocket push delivers the new value

**@MainActor serialization note:** `CameraState` is `@MainActor`. All `adjustToValue` calls execute sequentially on the main actor, even when wrapped in `withTaskGroup` or independent `Task {}`. True parallelism would require restructuring to run HTTP requests off-actor with a callback ‚Äî not currently implemented.

## WebSocket + Polling Architecture

Post-command flow (ESP32 side):
1. Dashboard sends POST command ‚Üí ESP32 receives
2. ESP32 sets `poll_state_now = true` and `poll_property_hint = "av"` (or appropriate)
3. Main poll loop sees `poll_state_now` ‚Üí waits 150ms ‚Üí polls single property via Camera API
4. Broadcasts new value over WebSocket immediately
5. Waits 500ms, then resumes normal polling cycle

Dashboard side:
1. After HTTP POST: waits 300ms (give camera time to process)
2. Adaptive poll: checks `getCurrentValue()` every 50ms
3. Exits poll when value changes OR after timeout (3000ms iris, 1500ms others)

## TSL Tally System

The dashboard receives TSL UMD protocol packets from video switchers (ATEM, etc.) and controls RGB LEDs on each ESP32 bridge.

### Architecture

```
Video Switcher ‚Üí TSL TCP (port 5201) ‚Üí Dashboard TSLClient ‚Üí Camera TSL Assignment ‚Üí ESP32 /api/tally ‚Üí RGB LED
```

### Key Files

| File | Purpose |
|------|---------|
| `TSLClient.swift` | TCP listener for TSL 3.1 and 5.0 protocols |
| `TallySettingsView.swift` | UI for TSL port config, camera assignments, LED brightness |
| `Camera.swift` | `tslIndices: [Int]` field, tally state, `updateTallyState()`, `sendBrightness()` |
| `CameraManager.swift` | TSL lifecycle, tally propagation, listening/connected state tracking |
| `ContentView.swift` | Tally border rendering on camera tiles |

### TSL Protocol Support

- **TSL UMD 3.1** (18-byte fixed format) ‚Äî most common
- **TSL UMD 5.0** (variable length) ‚Äî newer format
- **TCP only** (port 5201 configurable) ‚Äî UDP not supported
- Automatic protocol detection based on packet structure

### Camera Assignment

Each camera can be assigned one or more TSL indices (1-127) via a multi-select popover in Tally Settings (Cmd+Shift+T). When a TSL packet arrives with a matching index:
1. Dashboard matches index to all cameras with that index in their `tslIndices` array
2. Sends POST to `/api/tally/{state}` on each matching ESP32
3. ESP32 sets RGB LED at current brightness and broadcasts via WebSocket
4. Dashboard shows tally border (red=program, green=preview)

Multiple cameras can share the same TSL index (both will light up). Multiple indices per camera are supported (e.g., a camera on inputs 1 and 5 will respond to either).

**TSL state logic:** Program always wins over preview ‚Äî if program=true, "program" is sent regardless of preview state. This avoids the brief amber flash that occurs when some switchers (ATEM) momentarily send both program and preview true during a cut.

**LED brightness:** Global brightness slider (1‚Äì100%) in Tally Settings sends PWM value (0‚Äì255) to all ESP32s via `/api/tally/brightness/{value}`. Brightness is persisted in UserDefaults and restored when each ESP32 connects.

### LED Wiring

ESP32 GPIO pins for tally LEDs (controlled via LEDC PWM for brightness):
- **GPIO 1** ‚Äî Red LED (program)
- **GPIO 2** ‚Äî Green LED (preview)
- Standard 5mm LEDs with 220Œ© resistors to GND
- LEDC timer 0, channels 0 (red) and 1 (green), 8-bit resolution (0‚Äì255), 1000 Hz

### TSL Status (Three States)

The dashboard TSL indicator shows three states:
- **Gray** ‚Äî TSL disabled
- **Yellow** ‚Äî Listening (port bound, no switcher connected)
- **Green** ‚Äî Switcher connected and sending data

## Release Rule ‚Äî Every Change Must Ship

**Any change to the app OR firmware must go through a full release so the client can receive it via Sparkle OTA.**

This means every fix, no matter how small, requires:
1. Bump the version (ask first ‚Äî usually patch: 1.0.x)
2. Rebuild the app (`bash build.sh`) ‚Äî bundles the latest firmware automatically
3. Create a signed zip and GitHub release
4. Update the appcast (`app-updates/appcast-c200controller.xml`) and push it

**Why:** The client machine runs the installed app. If we fix something locally and don't release it, the client is still on the old version and can't get the fix. Sparkle is the only delivery path.

**Firmware changes specifically:** Firmware is bundled inside the app at build time. A firmware fix requires an app release too ‚Äî bump both the firmware version (`#define FIRMWARE_VERSION` in `main.c`) and the app version. The client then pushes firmware to boards via ‚åò‚áßU.

## Security Notes

This tool is **local-network-only** (closed church WiFi + Ethernet). Known by-design posture:

- **HTTP not HTTPS**: Canon C200 local API doesn't support HTTPS. All traffic stays on isolated production network.
- **No authentication on ESP32 endpoints**: Intentional for local trusted environment. Anyone on the network can send commands.
- **Camera credentials hardcoded**: `admin:admin` (Canon default) in Camera.swift. Not persisted ‚Äî change requires rebuild.
- **WiFi credentials in firmware**: Hardcoded in `main.c` during development; provisioned via ESP32Flasher tool for end-user units.
- **CORS wildcard** (`Access-Control-Allow-Origin: *`): Acceptable for local-only ‚Äî not internet-exposed.

Before any public/shared release: move credentials to Keychain (camera) and NVS/provisioning (WiFi).
